## Part 1: Current Setup on the Smartworks site (DONE)

### 1a. Posts via NimblePublisher

Straight from the canonical pattern, adapted. In `mix.exs`:

```elixir
{:nimble_publisher, "~> 1.1"},
{:makeup_elixir, ">= 0.0.0"},
{:makeup_erlang, ">= 0.0.0"},
```

A `Post` struct at `lib/my_app/blog/post.ex`:

```elixir
defmodule MyApp.Blog.Post do
  @enforce_keys [:id, :title, :body, :description, :tags, :date]
  defstruct [:id, :title, :body, :description, :tags, :date]

  def build(filename, attrs, body) do
    [year, month_day_id] = filename |> Path.rootname() |> Path.split() |> Enum.take(-2)
    [month, day, id] = String.split(month_day_id, "-", parts: 3)
    date = Date.from_iso8601!("#{year}-#{month}-#{day}")
    struct!(__MODULE__, [id: id, date: date, body: body] ++ Map.to_list(attrs))
  end
end
```

Your `Blog` context at `lib/my_app/blog.ex`:

```elixir
defmodule MyApp.Blog do
  alias MyApp.Blog.Post

  use NimblePublisher,
    build: Post,
    from: Application.app_dir(:my_app, "priv/posts/**/*.md"),
    as: :posts,
    highlighters: [:makeup_elixir, :makeup_erlang]

  @posts Enum.sort_by(@posts, & &1.date, {:desc, Date})
  @tags @posts |> Enum.flat_map(& &1.tags) |> Enum.uniq() |> Enum.sort()

  def all_posts, do: @posts
  def all_tags, do: @tags
  def get_post_by_id!(id), do: Enum.find(@posts, &(&1.id == id)) || raise("no post #{id}")
end
```

Posts live in `priv/posts/YYYY/MM-DD-slug.md` with the front-matter/body split that NimblePublisher expects. Wire up `/` (index), `/posts/:id` (show), and `/tags/:tag` through a normal controller — nothing fancy.

Add the live-reload pattern so markdown edits hot-reload in dev:

```elixir
# config/dev.exs
live_reload: [patterns: [..., ~r"priv/posts/.*(md)$"]]
```

### 1b. The export mix task

Put this at `lib/mix/tasks/my_app.export.ex`. Keep it close to your app for now — we'll extract later.

```elixir
defmodule Mix.Tasks.MyApp.Export do
  use Mix.Task

  @shortdoc "Export the site to priv/static_site"
  @output "priv/static_site"

  @impl Mix.Task
  def run(_args) do
    Mix.Task.run("app.start")

    File.rm_rf!(@output)
    File.mkdir_p!(@output)

    # 1. Copy compiled assets (digested or not, your call)
    copy_static_assets()

    # 2. Render every exportable route
    for path <- exportable_paths() do
      html = render(path)
      write(path, html)
    end

    # 3. Write sitemap
    File.write!(Path.join(@output, "sitemap.xml"), sitemap(exportable_paths()))

    Mix.shell().info("Exported to #{@output}")
  end

  defp exportable_paths do
    static_paths = ["/", "/about", "/tags"]
    post_paths = Enum.map(MyApp.Blog.all_posts(), &"/posts/#{&1.id}")
    tag_paths = Enum.map(MyApp.Blog.all_tags(), &"/tags/#{&1}")
    static_paths ++ post_paths ++ tag_paths
  end

  defp render(path) do
    Phoenix.ConnTest.build_conn()
    |> Phoenix.ConnTest.dispatch(MyAppWeb.Endpoint, :get, path)
    |> Phoenix.ConnTest.html_response(200)
  end

  defp write("/", html), do: File.write!(Path.join(@output, "index.html"), html)
  defp write(path, html) do
    dir = Path.join(@output, String.trim_leading(path, "/"))
    File.mkdir_p!(dir)
    File.write!(Path.join(dir, "index.html"), html)
  end

  defp copy_static_assets do
    src = Application.app_dir(:my_app, "priv/static")
    File.cp_r!(src, @output)
  end

  defp sitemap(paths) do
    base = "https://yoursite.com"
    urls = Enum.map_join(paths, "\n", &"<url><loc>#{base}#{&1}</loc></url>")
    ~s(<?xml version="1.0"?>\n<urlset xmlns="http://www.sitemaps.org/schemas/sitemap/0.9">\n#{urls}\n</urlset>)
  end
end
```

Two things worth noting because they trip people up:

- **You need Phoenix.ConnTest at runtime.** It normally lives in `:test` deps. Either move it to `{:phoenix, ...}` (it's already there transitively) or use `Plug.Test` + a direct endpoint call. The `ConnTest` approach is the cleanest — just make sure `:phoenix` isn't `only: :test`.
- **Run it against prod-compiled assets.** `MIX_ENV=prod mix assets.deploy && MIX_ENV=prod mix my_app.export`. Otherwise you ship unminified CSS with dev cachebusters.

## Part 2: Extract into a library (PENDING)

Once Part 1 is working, you'll have earned opinions about what belongs in the library. Here's how I'd scope it — and a few places I'd push back on your initial design.

### 2a. Name and shape

Call it something descriptive — `phoenix_ssg`. Structure:

```
phoenix_ssg/
├── lib/
│   ├── phoenix_ssg.ex              # config + main API
│   ├── phoenix_ssg/
│   │   ├── exporter.ex             # the render-and-write core
│   │   ├── route_discovery.ex      # which routes to export
│   │   ├── lint.ex                 # exportability checker
│   │   └── sitemap.ex              # sitemap generation
│   └── mix/tasks/
│       ├── phoenix_ssg.export.ex    # the export task
│       ├── phoenix_ssg.lint.ex      # the lint task
│       └── phoenix_ssg.install.ex   # Igniter installer
```

### 2b. Your three features, revisited

**Feature 1 — Igniter installer for `/sitemap.xml`.** This is fine, but I'd broaden the installer's job. A user running `mix igniter.install phoenix_ssg` probably wants the library to also:

- add config stubs to `config/config.exs` (output dir, base URL, static paths)
- add a `MyAppWeb.SitemapController` and route (Igniter has `add_scope/4` for this — confirmed from the docs above)
- optionally add a `robots.txt` controller

The sitemap route itself is a thin controller that calls into your library's `Sitemap.generate/1`. Users typically want the sitemap reachable *in dev* too (so they can curl it, so crawlers hit it if they deploy the Phoenix app itself), hence making it a real route rather than only a build-time file.

**Feature 2 — the export mix task.** This is the main event. Configuration shape I'd suggest:

```elixir
# config/config.exs
config :phoenix_ssg,
  endpoint: MyAppWeb.Endpoint,
  output_dir: "priv/static_site",
  base_url: "https://yoursite.com",
  # Extra paths not discoverable from the router (dynamic shows)
  extra_paths: {MyApp.Blog, :all_post_paths, []},
  # Paths to skip even if they look exportable
  exclude: ["/dev/*", "/health"]
```

The `extra_paths` MFA is the key design point. Your router knows about `/posts/:id` but it doesn't know which IDs to render. Users need a way to say "call this function to get the concrete paths." Keep it simple: accept `{module, fun, args}` that returns a list of strings.

**Feature 3 — the exportability linter.** This is the feature I'd push back on hardest, and I want to be direct about why: detecting "pages with forms" statically is a leaky proxy for "pages that can't be static." You can have:

- forms that submit to external services (fine to export)
- LiveViews that degrade gracefully without JS (fine-ish)
- static-looking pages that rely on session data (broken when static, but no form)
- pages behind pipelines that set cookies (same problem, invisible to AST scan)

Rather than trying to detect non-exportability by scanning templates, I'd invert it: **render every route and flag anomalies at export time**. Things you can detect reliably at render time:

- non-200 responses (redirects, 500s)
- `<form method="post">` tags in output (real signal)
- presence of CSRF tokens (the page assumes a session)
- response size anomalies (error pages are often tiny)
- responses that set session cookies

So `mix phoenix_ssg.lint` would essentially do a dry-run export and report findings, rather than doing AST analysis. Much more accurate, much simpler code.

### 2c. Igniter installer sketch

```elixir
defmodule Mix.Tasks.PhoenixStaticExport.Install do
  use Igniter.Mix.Task

  @impl Igniter.Mix.Task
  def info(_argv, _source), do: %Igniter.Mix.Task.Info{group: :phoenix_ssg}

  @impl Igniter.Mix.Task
  def igniter(igniter) do
    igniter
    |> Igniter.compose_task("igniter.add_extension", ["phoenix"])
    |> add_config()
    |> add_sitemap_controller()
    |> add_sitemap_route()
  end

  defp add_sitemap_route(igniter) do
    Igniter.Libs.Phoenix.add_scope(igniter, "/", """
    get "/sitemap.xml", SitemapController, :show
    """, arg2: Igniter.Libs.Phoenix.web_module(igniter))
  end

  # ...
end
```

`Igniter.Libs.Phoenix.add_scope/4` is the right primitive here — confirmed from the current `igniter v0.6.28` docs.

### 2d. Route discovery via `Phoenix.Router.routes/1`

For feature 3 (and also to drive feature 2), you want to enumerate routes from the router. Phoenix exposes this cleanly:

```elixir
def discoverable_paths(router) do
  router.__routes__()
  |> Enum.filter(&(&1.verb == :get))
  |> Enum.reject(&dynamic?(&1.path))
  |> Enum.map(& &1.path)
end

defp dynamic?(path), do: String.contains?(path, ":") or String.contains?(path, "*")
```

Combine with the user's `extra_paths` MFA to get the full list. This also gives you a natural place to warn: "Your router has `/posts/:id` but nothing in `extra_paths` renders it — these will be missed."
